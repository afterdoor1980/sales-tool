## 二、按内容性质划分的五种模板类型

### 类型A：话术库类
适用范围：异议处理、需求挖掘、成交逼单、破冰开场等

| 字段 | 说明 |
|---|---|
| 场景/触发条件 | 客户说了什么话，或处于什么情境。**应尽量记录客户的真实模糊表述（如"我再考虑一下"），而非提炼后的标准化问法（如"客户提出预算异议"）**，因为现场遇到的往往是模糊或复合表述，标准化问法会让使用者难以对应到具体场景 |
| 对话对象角色 | 决策人 / 使用者 / 财务把关人（B2B分角色关键字段） |
| 标准话术 | 推荐说法 |
| 禁忌话术 | 不能说什么，常见踩坑点 |
| 话术意图（四维度） | 见下方《话术意图四维度填写规范》，不再是单一自由文本，而是四个子问题必答 |
| 升级路径 | 若说完标准话术后客户仍不接受（二次追问、坚持原立场），下一步该怎么办：是否有第二轮应对方向，还是应转交更高层级/其他角色处理，转交的具体触发条件是什么（见下方《升级路径填写规范》） |
| 使用注意 | 语气、时机等补充说明 |

#### 升级路径填写规范

每条标准话术都只是"第一轮"应对，客户不接受是常态而非例外。若模板中该字段留空或简单写"继续沟通"，等于把二轮应对完全丢给一线人员临场发挥，抵消了话术库本身的价值。填写时需明确以下两种情况之一（或都有）：

| 情况 | 填写要求 |
|---|---|
| 存在可预期的第二轮应对逻辑 | 简要说明第二轮该往哪个方向调整（如从"价值论证"转为"提供有条件让步"），不需要重复完整话术，指出方向和边界即可 |
| 需要转交更高层级或其他角色处理 | 明确转交条件（客户说了什么/达到什么程度就该转交，而非"感觉搞不定了"这种主观判断），以及转交给谁（如超出本人报价权限转交上级，技术争议转交技术把关人对接） |

#### 话术意图四维度填写规范

单一自由文本的"意图说明"容易只写表层理由、漏掉深层考虑。填写时必须依次回答以下四个子问题，帮助撰写者（尤其是共创现场的一线销售人员）做更全面的思考，而非凭感觉写一句话：

| 思考维度 | 引导问题 |
|---|---|
| 情绪维度 | 客户此刻是什么情绪（防御/焦虑/试探）？这句话想把他的情绪引向哪里？ |
| 信息目标维度 | 我想从客户那里换取什么信息，或者想让客户获得什么信息？ |
| 关系维度 | 这句话会不会损害长期信任（比如过度承诺、打价格锚点）？ |
| 行动导向维度 | 说完这句话，期望客户下一步做什么具体动作（而非泛泛的"觉得满意"）？ |

#### 异议识别与拆解（类型A话术库使用前置指南）

客户的真实表述往往模糊、混合，或是委婉说法，不会自报"我这是价格异议"。在套用具体异议类型话术前，需先做识别与拆解，避免误判类型导致话术答非所问。

**常见模糊/伪装表述识别原则**：同一句话（如"我再考虑一下"）可能对应多种真实异议类型（预算延迟/信任顾虑/单纯敷衍），使用时应先用一句澄清提问确认真实类型，再选用对应话术，而非见到关键词就直接套话术。

**复合异议处理规则**（客户一句话同时包含多类信号时）：
1. 先处理占比更大/更根本的那类（可参考情绪强度判断，如陌生信任感通常比价格更根本）
2. 用一句过渡语搭桥，再进入第二类话术，不要在一次回应里堆砌两类完整话术
3. 避免让客户感觉在被"话术攻势"应对，而非被认真倾听

各模块在类型A话术库正文前，应视需要增加一节"该场景下的模糊表述对照表"，列出常见原始表述、可能对应的真实类型、以及澄清提问，具体示例见 S11 模块。



#### 条目编号规则

字段规范多处提到类型间可互相引用（如类型D的执行动作引用类型A话术），此前只说明"需要编号"但未定义具体格式，统一规则如下：

- **格式**：`{模块编号}-{两位序号}`，如 `S11-01`、`S11-02`，按该条目在模块正文中出现的先后顺序编号
- **编号本身不编码案例/类型/角色信息**（不做成如"S11-甲-价格-01"这类复合编码），具体内容由条目本身的字段呈现，编号仅作引用时的定位锚点，保持简洁、不因内容调整而需要重新编码
- **跨模块引用时**，写明"参照 {编号} 的{具体字段}"，如"执行动作：参照S11-02升级路径处理"，而非用标题文字描述去定位
- 新增/删除条目导致后续编号变动时，需同步检查其他模块中对该编号的引用是否需要更新

#### 角色分支规则（同一场景对不同角色是否需要拆分话术）

B2B决策链常涉及多个角色（如案例公司甲的采购经理/老板，案例公司乙的技术把关人/财务/分管领导等），同一异议或场景在不同角色面前，应对逻辑可能不同，但不能无限拆分导致组合爆炸，需遵循以下规则：

| 分支规则 | 说明 |
|---|---|
| 只在"角色关注点实质不同"时才拆分 | 若不同角色面对同一场景的应对逻辑本质相同（仅语气正式程度有别），不拆分；若本质不同（如价格异议对采购经理谈成本对比、对老板谈战略合作价值），才拆出对应角色的独立条目 |
| 拆分优先级给高价值/高频角色组合 | 决策链角色较多时（如案例公司乙），不需要每类场景覆盖全部角色，优先覆盖"最常引发该场景/异议的角色"，其余角色若无实质差异则不单独列出 |

### 类型B：判定/分类标准类
适用范围：客户分层标准、优质客户判定、方案设计标准等

| 字段 | 说明 |
|---|---|
| 判定维度 | 如"预算规模""决策周期" |
| 判定取值/标准 | 具体数值区间或描述性标准 |
| 对应等级/结论 | S/A/B/C 或 "合格/不合格" |
| 该等级对应动作 | 分层/判定后具体该如何跟进，避免"只分层不指导动作" |

#### 判定状态区分原则（"存疑"与"信息不足"不可混用）

类型B模块在设计判定标准时，需区分以下两种容易混淆的状态，处理方式不同：

| 状态 | 含义 | 处理方向 |
|---|---|---|
| 存疑 | 已掌握信息，且信息本身是负面信号 | 需谨慎处理或转交上级评估风险 |
| 信息不足 | 尚未掌握足够信息，无法判断正面或负面（不是负面信号，只是还不知道） | 暂缓判定，通过后续接触补充信息，不直接转交上级，也不因此放弃投入 |

同时，判定结论若带有负面标签（如"疑似比价""疑似陪标"），应给出**重新评估触发条件**，说明对象出现何种转变信号时应重新按当前标准判定，避免标签定型后不再更新。

若某类判定中多个维度同时处于"信息不足"状态（而非全部维度不合格），应给出中间地带的处理动作（如"低资源投入观察"），而非强行判定合格或不合格。



### 类型C：流程/清单类
适用范围：拜访前准备、获客渠道方法、售后服务流程等

| 字段 | 说明 |
|---|---|
| 步骤序号 | 流程顺序 |
| 具体动作 | 要做什么 |
| 责任人/角色 | 谁执行 |
| 产出物/检查点 | 完成标志，便于后续验收 |

### 类型D：节奏/触发规则表类

类型D下细分两种子模式，字段结构不同，使用前需先判断该模块属于哪一种，不可混用：

**子模式D1：周期节奏型**
适用范围：跟进节奏表等——核心逻辑是"多久一次、到了时间点做什么"，关键变量是时间

| 字段 | 说明 |
|---|---|
| 触发条件 | 什么情况下启动这条规则 |
| 对应时间节点/频率 | 具体周期或时点 |
| 执行动作 | 做什么（可关联类型A的具体话术编号） |
| 责任人 | 谁执行 |

**子模式D2：阶梯判定型**
适用范围：阶梯报价与让步节奏表等——核心逻辑是"客户/情况达到什么程度、对应哪一级处理动作、谁能拍板"，关键变量是程度/等级而非时间

| 字段 | 说明 |
|---|---|
| 阶梯层级 | 第几级（如首次报价、第一次让步、底线等） |
| 触发条件 | 什么情况下进入这一级（达到什么程度/第几次提出要求等） |
| 处理幅度 | 该级别具体的让步/处理幅度区间，需给出可执行数字或范围，不能是模糊描述 |
| 审批权限 | 该级别谁能拍板（一线自主/需主管审批/需更高层审批） |
| 执行动作/话术引用 | 具体怎么做，可关联类型A的具体话术编号 |
| 使用注意 | 补充说明，包括是否受客户分层影响（如涉及，标注占位待S04对齐） |

**判断依据**：若该模块的核心问题是"多久该做一次什么"，用D1；若核心问题是"事情严重/客户要求到什么程度，该给到哪一级处理、谁批"，用D2。当前标注为类型D的模块中，S14（跟进节奏）为D1；S10（阶梯报价）确认为D2；S15、S19暂标注为待确认，需在具体设计时先判断子模式再套用字段，避免生搬硬套导致结构别扭。

### 类型E：案例萃取类
适用范围：销冠案例萃取、成功案例、失败案例复盘

| 字段 | 说明 |
|---|---|
| 案例背景 | 客户情况、业务模式（甲/乙） |
| 关键决策点 | 谈单过程中的转折点/难点 |
| 销冠动作/话术 | 具体怎么做的（可引用类型A话术编号） |
| 结果 | 成交/失败及原因 |
| 可复制规律 | 从个案抽象出的通用方法论，是把"个人经验"转化为"组织知识"的关键字段 |

## 三、设计要点

1. **类型间可相互引用**：例如类型D的"执行动作"可直接引用类型A的具体话术编号，字段之间形成勾稽关系，而非互相独立的孤岛。因此所有类型A话术条目必须有唯一编号。
2. **类型B新增"该等级对应动作"字段**：修正常见分层"只贴标签、不指导动作"的问题。
3. **类型选择优先于内容填写**：后续每个模块的具体模板设计，遵循"先选类型、再套字段、最后填内容"的流程，不再逐模块重新设计结构，以保证效率与结构一致性。
4. **企业替换案例内容的通用操作说明**：见S01模块文档"企业如何将示范内容替换为自己真实信息"一节，该说明为通用性质，后续所有以案例公司甲/乙为示范的模块均可直接引用，不需要逐模块重复撰写。
